## 1. 概念模型

```
users 1 ── * research_message_reads
research_messages 1 ── * research_message_reads
```

- **消息**全局可见范围由 **`visibility`** / **`tenant_id`**（若多租户）控制；P0 可简化为「登录用户可见全部消息」。  
- **已读**按 **用户 × 消息** 存一条记录。

---

## 2. 表定义（PostgreSQL 建议）

表前缀 `research_`；团队可改为 `m1_`，**字段语义不变**。

### 2.1 `research_messages` — 投研消息主表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | `UUID` | PK | 即 `messageId` |
| `title` | `VARCHAR(500)` | NOT NULL | |
| `summary` | `TEXT` | NULL | 列表摘要 |
| `body` | `TEXT` | NOT NULL | 详情正文 |
| `content_format` | `VARCHAR(16)` | NOT NULL，默认 `markdown` | `markdown`/`html`/`plain` |
| `published_at` | `TIMESTAMPTZ` | NOT NULL | 排序主键之一 |
| `category` | `VARCHAR(64)` | NULL | 筛选 |
| `source_type` | `VARCHAR(32)` | NULL | **预留** M2/Mock |
| `source_name` | `VARCHAR(128)` | NULL | 展示 |
| `source_system` | `VARCHAR(64)` | NULL | **预留**：写入系统标识（如 `glue-v1`），M1 开发期可空 |
| `external_ref` | `VARCHAR(256)` | NULL | **预留**：上游主键，防重复 ingest（唯一性由后续 M2 设计，M1 可先不建 UNIQUE） |
| `links` | `JSONB` | NULL | `[{ "label", "url" }]` |
| `metadata` | `JSONB` | NULL | **扩展桶**，M4/M5 等可挂角标、trace 提示；M1 不解析 |
| `search_vector` | `tsvector` | NULL | 可选，全文检索用 |
| `deleted_at` | `TIMESTAMPTZ` | NULL | 软删；列表/搜索默认 `IS NULL` |
| `created_at` | `TIMESTAMPTZ` | NOT NULL，默认 `now()` | |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL，默认 `now()` | |

**索引**

- `(published_at DESC, id DESC)` — 列表与游标分页。  
- `(category)` WHERE `deleted_at IS NULL`。  
- `GIN(search_vector)` — 若启用全文检索。  
- 可选 `(source_system, external_ref)` UNIQUE — **在 M2 接入时**再加，避免过早锁死。

### 2.2 `research_message_reads` — 用户已读

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `user_id` | `UUID` | PK 复合 | FK → `users.id` |
| `message_id` | `UUID` | PK 复合 | FK → `research_messages.id` ON DELETE CASCADE |
| `read_at` | `TIMESTAMPTZ` | NOT NULL，默认 `now()` | |

**唯一约束**：`(user_id, message_id)`。

**索引**：`(user_id, read_at DESC)` — 「我的已读历史」扩展用。

### 2.3 列表「未读」判定

对当前 `user_id`，`read` = 存在 `research_message_reads` 行。  
`unreadOnly=true`：`NOT EXISTS (SELECT 1 FROM research_message_reads r WHERE r.user_id = ? AND r.message_id = m.id)`。

## 3. 搜索实现（MVP → 演进）

| 阶段 | 实现 | 说明 |
|------|------|------|
| MVP | `title`/`summary`/`body` 的 `ILIKE %q%` 或 `pg_trgm` | 简单，数据量小时可接受 |
| P1 | 维护 `search_vector`，`to_tsvector('simple', coalesce(title,'') \|\| body)` | 中文需额外配置或外部索引 |

`GET /research/messages/search` 的 `highlight` 可由应用层从命中片段截取。

---

## 4. 扩展预留（不接 M2/M4 时如何做）

| 场景 | 做法 |
|------|------|
| 独立开发 M1 | Migration 插入 **种子数据** 若干条；或管理脚本 `scripts/seed_research_messages.py` |
| 字段空置 | `source_system`、`external_ref`、`metadata` 保持 `NULL` / `{}` |
| 接入 M2 后 | 由 M2 作业 **INSERT/UPDATE** `research_messages`（或经中间表同步）；本文表结构 **尽量不 breaking** |

---

## 5. 存储与合规

| 项 | 建议 |
|----|------|
| RDS | PostgreSQL，可与全项目共享实例 |
| 正文过大 | P0 单条 `body` 建议 **≤ 512KB**；超出走 OSS + `body` 存摘要+链接（P1） |
| 留存 | 与合规一致；软删优先，物理删除异步任务 |
| PII | `body` 内可能含敏感信息；日志 **禁止** 打全文 |
